# BOOK_PROTOCOL.md

**Canonical rule for structuring long-form deliverables as multi-page books.**

Every documentation-producing agent references this file. It is the single source of truth for when and how to split a deliverable into a book.

---

## The Rule

> Any deliverable expected to exceed **300 lines** MUST be structured as a book: a directory of chapter files with a `README.md` index. Single-file deliverables are only acceptable when the output will stay under 300 lines.

---

## Book Structure

```
docs/<deliverable-slug>/
├── README.md          ← index: title, date, Part I / II / III table with links
├── 01-<topic>.md      ← chapter 1
├── 02-<topic>.md      ← chapter 2
├── ...
└── A-appendices.md    ← optional appendix
```

### Naming rules

- Directory slug: lowercase, hyphens, no spaces — e.g. `security-audit`, `code-review`, `architecture`
- Chapter files: zero-padded number + hyphen + slug — `01-overview.md`, `02-findings.md`
- Appendices: letter prefix — `A-appendices.md`, `B-glossary.md`
- Index: always `README.md`

---

## README.md Template

```markdown
# <Deliverable Title>

**Date:** YYYY-MM-DD | **Version:** vX.Y.Z | **Author:** <agent name>

One-sentence description of what this book covers.

---

## Part I — <Group Name>

| Chapter | Summary |
|---------|---------|
| [01 · Topic](01-topic.md) | One-line description |
| [02 · Topic](02-topic.md) | One-line description |

## Part II — <Group Name>

| Chapter | Summary |
|---------|---------|
| [03 · Topic](03-topic.md) | One-line description |

---

*Generated by <agent-name> on <date>.*
```

---

## Chapter File Template

Each chapter file starts and ends with a navigation bar:

```markdown
[🏠 Index](README.md)  |  [← Previous Title](prev-file.md)  |  [Next Title →](next-file.md)

---

# <Chapter Title>

<content>

---

[🏠 Index](README.md)  |  [← Previous Title](prev-file.md)  |  [Next Title →](next-file.md)
```

- First chapter: omit the `← prev` link
- Last chapter: omit the `next →` link
- Index link is always present on every chapter

---

## Grouping guidance

Group chapters into Parts by theme, not by how they were generated. Typical groupings:

| Report type | Part I | Part II | Part III |
|-------------|--------|---------|---------|
| Security audit | Scope & Methodology | Findings | Remediation |
| Code review | Architecture | Per-module findings | Recommendations |
| Onboarding | System overview | Component inventory | Gaps & next steps |
| Performance | Baseline measurements | Bottlenecks | Improvement plan |
| Research | Executive summary | Deep findings | Sources |
| Architecture | Overview & principles | Module design | Infrastructure |
| SDLC improve | Discovery & scope | Audit findings | Improvement backlog |

Aim for 3–6 chapters per Part. If a Part would have only 1 chapter, merge it into an adjacent Part.

## Validation

After writing all book files, run:

```bash
bash scripts/validators/validate-book-structure.sh docs/<deliverable-slug>/
bash scripts/validators/validate-mermaid.sh . docs/<deliverable-slug>/
```

Both must exit 0 before the deliverable is marked DONE.

`validate-book-structure.sh` checks:
- `README.md` exists and has a navigation table
- At least 2 chapter files exist
- Every chapter file has nav bars (top + bottom)
- No chapter file exceeds 400 lines

---

## Single-file exception

A deliverable stays as a single file when:
- It is reliably under 300 lines (short audit summary, single-dimension report)
- It is an intermediate work file, not a final deliverable (`docs/work/`)
- It is a tracker (`SDLC_TRACKER.md`, `DELEGATION_LOG.md`)
- It is a Completion Manifest

When in doubt — split. A 200-line book with 3 chapters is easier to navigate than a 200-line monolith.

---

## Why books, not monoliths

- **Context window pressure**: a 1,000-line document loaded whole consumes ~4k tokens. Three 300-line chapters loaded on demand cost 1k tokens each.
- **Navigation**: readers jump to the chapter they need; they don't scroll.
- **Incremental writing**: a chapter can be written, verified, and committed without holding the full document in context.
- **Validators**: `validate-mermaid.sh` and `validate-book-structure.sh` scan faster on small files and report errors with precise line numbers.
