# BOOK_PROTOCOL.md

**Canonical rule for structuring long-form deliverables as multi-page books.**

Every documentation-producing agent references this file. It is the single source of truth for when and how to split a deliverable into a book.

---

## The Rule

> Any deliverable expected to exceed **300 lines** MUST be structured as a book: a directory of chapter files with a `README.md` index. Single-file deliverables are only acceptable when the output will stay under 300 lines.

> Any chapter expected to exceed **400 lines** MUST itself be split into a chapter directory with sub-chapter files. Maximum nesting depth is **2 levels** (book → chapter-dir → sub-pages). Never go deeper.

---

## Book Structure

### Flat chapters (chapters stay under 400 lines)

```
docs/<deliverable-slug>/
├── README.md          ← book index: title, date, Part table with links
├── 01-<topic>.md      ← chapter 1 (flat file)
├── 02-<topic>.md      ← chapter 2 (flat file)
├── ...
└── A-appendices.md    ← optional appendix
```

### With sub-chapters (a chapter exceeds 400 lines → becomes a directory)

```
docs/<deliverable-slug>/
├── README.md              ← book index
├── 01-<topic>.md          ← short chapter, stays flat
├── 02-<topic>/            ← long chapter → becomes a directory
│   ├── README.md          ← chapter index (links to sub-pages)
│   ├── 01-<sub>.md        ← sub-chapter 1
│   ├── 02-<sub>.md        ← sub-chapter 2
│   └── 03-<sub>.md        ← sub-chapter 3
├── 03-<topic>.md          ← another short chapter, stays flat
└── A-appendices.md
```

**The same 400-line rule applies at both levels.** A chapter file that grows past 400 lines becomes a chapter directory. A sub-chapter file that grows past 400 lines is a sign the content needs consolidation, not another nesting level.

---

## Naming rules

### Book level
- Directory slug: lowercase, hyphens, no spaces — e.g. `security-audit`, `code-review`
- Chapter files: zero-padded number + hyphen + slug — `01-overview.md`, `02-findings.md`
- Chapter directories: same pattern — `02-findings/`
- Appendices: letter prefix — `A-appendices.md`, `B-glossary.md`
- Index: always `README.md`

### Chapter directory level
- Sub-chapter files: same numbering scheme — `01-auth.md`, `02-injection.md`
- Index: `README.md` (the chapter landing page / intro)
- No further sub-directories — 2-level cap is absolute

## README.md Template

### Book-level README

```markdown
# <Deliverable Title>

**Date:** YYYY-MM-DD | **Version:** vX.Y.Z | **Author:** <agent name>

One-sentence description of what this book covers.

---

## Part I — <Group Name>

| Chapter | Summary |
|---------|---------|
| [01 · Topic](01-topic.md) | One-line description |
| [02 · Topic](02-topic/) | One-line description (links to directory → README.md resolves) |

## Part II — <Group Name>

| Chapter | Summary |
|---------|---------|
| [03 · Topic](03-topic.md) | One-line description |

---

*Generated by <agent-name> on <date>.*
```

### Chapter-directory README

```markdown
# <Chapter Title>

[🏠 Book Index](../README.md)

Brief intro to what this chapter covers.

---

| Section | Summary |
|---------|---------|
| [01 · Sub-topic](01-sub.md) | One-line description |
| [02 · Sub-topic](02-sub.md) | One-line description |
```

## Nav bar formats

### Flat chapter (1 breadcrumb)

```markdown
[🏠 Index](README.md)  |  [← Previous Title](prev-file.md)  |  [Next Title →](next-file.md)
```

- First chapter: omit `← prev` link
- Last chapter: omit `next →` link
- Index link is always present

### Sub-chapter (2 breadcrumbs)

```markdown
[🏠 Book](../README.md)  |  [📖 Chapter](README.md)  |  [← Previous](prev.md)  |  [Next →](next.md)
```

- `[🏠 Book]` → book-level `README.md` (one level up: `../README.md`)
- `[📖 Chapter]` → chapter directory's `README.md` (same level: `README.md`)
- First sub-chapter: omit `← prev`; last sub-chapter: omit `next →`

Place the nav bar at the **top and bottom** of every chapter and sub-chapter file.

## Grouping guidance

Group chapters into Parts by theme, not by how they were generated. Typical groupings:

| Report type | Part I | Part II | Part III |
|-------------|--------|---------|---------|
| Security audit | Scope & Methodology | Findings | Remediation |
| Code review | Architecture | Per-module findings | Recommendations |
| Onboarding | System overview | Component inventory | Gaps & next steps |
| Performance | Baseline measurements | Bottlenecks | Improvement plan |
| Research | Executive summary | Deep findings | Sources |
| Architecture | Overview & principles | Module design | Infrastructure |
| SDLC improve | Discovery & scope | Audit findings | Improvement backlog |

Aim for 3–6 chapters per Part. If a Part would have only 1 chapter, merge it into an adjacent Part.

When to use a chapter directory: if a single findings chapter would have 8+ sub-topics, split it into a `02-findings/` directory rather than one 600-line flat file.

## Validation

After writing all book files, run:

```bash
bash scripts/validators/validate-book-structure.sh docs/<deliverable-slug>/
bash scripts/validators/validate-mermaid.sh . docs/<deliverable-slug>/
```

Both must exit 0 before the deliverable is marked DONE.

`validate-book-structure.sh` checks:
- Book `README.md` exists and has a navigation table
- At least 2 chapter entries (files or directories) exist
- Every flat chapter file has nav bars and is ≤400 lines
- Every chapter directory has `README.md` with nav table
- Every sub-chapter has nav bars (both book and chapter breadcrumbs) and is ≤400 lines
- No 3rd-level nesting (warns if detected)

## Single-file exception

A deliverable stays as a single file when:
- It is reliably under 300 lines (short audit summary, single-dimension report)
- It is an intermediate work file, not a final deliverable (`docs/work/`)
- It is a tracker (`SDLC_TRACKER.md`, `DELEGATION_LOG.md`)
- It is a Completion Manifest

When in doubt — split. A 200-line book with 3 chapters is easier to navigate than a 200-line monolith.

---

## Why books, not monoliths

- **Context window pressure**: a 1,000-line document loaded whole consumes ~4k tokens. Three 300-line chapters loaded on demand cost 1k tokens each.
- **Navigation**: readers jump to the chapter they need; they don't scroll.
- **Incremental writing**: a chapter can be written, verified, and committed without holding the full document in context.
- **Validators**: `validate-mermaid.sh` and `validate-book-structure.sh` scan faster on small files and report errors with precise line numbers.
